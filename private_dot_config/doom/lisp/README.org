#+TITLE: Local Doom Lisp Modules

* Overview

This directory contains local Emacs Lisp modules loaded from Doom config.

The newest addition is =my-task-workflow.el=, which adds a task-centric Org
workflow for:

- planning in Org
- GitHub issue creation or linking via =gh=
- optional per-task git worktrees
- optional per-task Doom workspaces
- Codex handoff from the canonical main repository
- resuming work from the Org task subtree

The design is intentionally small. The Org task heading is the anchor, and the
rest of the state is stored as properties on that heading.

* Task Workflow

** File

- [[file:my-task-workflow.el][my-task-workflow.el]]

** Purpose

This module connects one Org task subtree to:

- a repository
- a GitHub issue
- a git branch
- an optional git worktree
- an optional Doom workspace

This lets the Org task act as the entry point for planning and resuming work.

** Expected Org shape

The workflow assumes a task subtree like this:

#+begin_src org
,* Basis operator updates

,** Basis operator phase 1
:PROPERTIES:
:REPO_PATH: ~/Documents/openghg_inversions
:GH_REPO: openghg/openghg_inversions
:GH_ISSUE: 417
:GH_URL: https://github.com/org/repo/issues/417
:BRANCH: issue-417-basis-operator-phase-1
:WORKTREE: ~/Documents/openghg_inversions-wt/openghg_inversions-iss-417
:WORKSPACE: openghg_inversions-iss-417
:END:

,*** Notes
Some implementation notes.

,*** user prompt
...

,*** AI response
...
#+end_src

The =WORKTREE= and =WORKSPACE= properties are optional and appear after a
worktree is adopted or an Emacs worktree/workspace is opened explicitly.

By default the task heading level is =2=, but this is configurable via
=my/task-heading-level=. Commands can be run from anywhere inside the task
subtree; they climb to the nearest ancestor heading at that level.

** Main commands

*** =M-x my/task-init=

Use this when you want to record task metadata without creating a GitHub issue,
worktree, or workspace.

It will:

1. find the task root
2. resolve =REPO_PATH= from the task, inherited Org properties, or an interactive prompt
3. derive and store metadata such as =BRANCH=, =GPTEL_TOPIC=, and =CREATED=
4. leave =WORKTREE= and =WORKSPACE= unset unless they already exist

This is the default entry point for recording a task without deciding yet
whether Emacs or Codex should own the worktree.

*** =M-x my/task-create=

Use this when a task does not yet have a GitHub issue.

It will:

1. find the task root
2. resolve =REPO_PATH= from the task, inherited Org properties, or an interactive prompt
3. export the direct =Notes= child subtree to Markdown
4. create a GitHub issue with =gh issue create=
5. store =GH_ISSUE= and =GH_URL=
6. derive and store metadata such as =BRANCH= and =GPTEL_TOPIC=

It does not create or open a worktree automatically.

*** =M-x my/task-link-issue=

Use this when the issue already exists on GitHub and you just want to attach it
to the current Org task.

It prompts for either:

- an issue number
- a full GitHub issue URL

Then it stores =GH_REPO=, =GH_ISSUE=, and =GH_URL=. Bare issue numbers are
expanded to full GitHub issue URLs from the task's repo context. Existing branch,
workspace, and worktree properties are preserved.

*** =M-x my/task-init-from-branch=

Use this when there is already a local branch and you want the task to take it
over.

It will:

1. prompt for an existing local branch in the repo
2. store that branch in =BRANCH=
3. leave =WORKTREE= and =WORKSPACE= unset

This is useful when a branch was started elsewhere and you want to bind it to an
Org task after the fact.

*** =M-x my/task-open-codex=

Use this when you want Codex to own the implementation worktree.

It will:

1. initialize task metadata if needed
2. resolve the primary worktree for =REPO_PATH=
3. open Codex from that main repo path
4. display and copy a short handoff prompt with the task, repo-plan location,
   issue URL, and suggested branch

Opening Codex from the main repo keeps Codex-created worktrees under the same
Codex project. If Codex creates a worktree, attach it back to the task later
with =my/task-adopt-worktree=.

*** =M-x my/task-adopt-worktree=

Use this when a worktree already exists, including a Codex-created worktree
under =~/.codex/worktrees=.

It will:

1. read =git worktree list --porcelain= from =REPO_PATH=
2. prompt for a registered worktree
3. store the selected path in =WORKTREE=
4. update =BRANCH= to the selected worktree's branch when it has one

It does not create or switch to a Doom workspace.

*** =M-x my/task-open=

Use this when you explicitly want an Emacs worktree/workspace context.

It will:

1. find the task root
2. read task properties
3. use an existing =WORKTREE= if one is set
4. adopt the only registered worktree for =BRANCH= if there is exactly one
5. otherwise create the derived worktree path
6. create or switch to the Doom workspace
7. open the worktree directory
8. open =magit-status= in that worktree

*** =M-x my/task-insert-skeleton=

Insert a small level-2 task skeleton at point with a =CREATED= timestamp,
derived =GPTEL_TOPIC=, and child headings for notes, acceptance criteria,
prompt, and outcome.

*** =M-x my/task-report-active-repo-tasks=

Show a lightweight report of active repo-plan tasks from =repo-plans/*.org=.
The report lists the file, line, TODO state, title, issue/PR, branch, and
=NEXT_ACTION= when present.

*** Property and link helpers

- =M-x my/task-describe-properties= shows direct, inherited, file-level, and derived values for managed task properties.
- =M-x my/task-set-property= sets a managed task property with completion.
- =M-x my/task-delete-property= deletes a direct managed task property.
- =M-x my/task-insert-gh-link= inserts an explicit =gh:owner/repo#123= link using the current task's =GH_REPO= or resolved repo context.
- =M-x my/task-add-worktree-to-gptel-allowed-roots= appends the task =WORKTREE= to the current gptel topic's =GPTEL_ALLOWED_ROOTS=.
- =M-x my/task-open-worktree-codex-dir= creates and opens =WORKTREE/.codex= for worktree-local Codex context.

** Naming rules

If =GH_ISSUE= exists:

- workspace: =<repo>-iss-<issue>=
- branch: =issue-<issue>-<slug>=

If =GH_ISSUE= does not exist:

- workspace: =<repo>-task-<slug>=
- branch: =<slug>=

Worktrees use a dedicated root per repository:

- worktree root: =<parent>/<repo>-wt=
- worktree path: =<worktree-root>/<workspace>=

Example:

- repo: =~/Documents/openghg_inversions=
- workspace: =openghg_inversions-iss-417=
- worktree: =~/Documents/openghg_inversions-wt/openghg_inversions-iss-417=

These names are materialized by =my/task-open=. Metadata-only commands do not
write =WORKTREE= or =WORKSPACE=.

** Properties

The workflow reads and writes these Org properties on the task heading:

- =REPO_PATH=
- =DEFAULT_BASE_BRANCH=
- =GH_REPO=
- =GH_ISSUE=
- =GH_URL=
- =PR=
- =BRANCH=
- =WORKTREE=
- =WORKSPACE=
- =GPTEL_TOPIC=
- =NEXT_ACTION=
- =CREATED=
- =LAST_REVIEWED=

=REPO_PATH= and =DEFAULT_BASE_BRANCH= may also be provided once per file using:

#+begin_src org
#+PROPERTY: REPO_PATH ~/Documents/openghg_inversions
#+PROPERTY: DEFAULT_BASE_BRANCH devel
#+end_src

When creating a new branch, the task workflow checks that the source repo is on
the configured base branch, or one of =main=, =devel=, =develop=, or =master= if
no explicit default is configured. If the branch has an upstream and is behind
it, branch creation is refused.

** Assumptions

- =gh= is installed and authenticated
- =git= is available
- Doom workspaces are enabled
- Magit is available
- task planning happens in Org files, possibly outside the repository

If a task is private or already has an issue elsewhere, use =my/task-init= or
=my/task-link-issue= instead of =my/task-create=.

* gptel Files

** [[file:my-gptel-tools.el][my-gptel-tools.el]]

Local filesystem and buffer tools for gptel.

The key idea is controlled access:

- read/search tools for local files and buffers
- optional edit tools
- write-root enforcement so edits stay inside an explicit repo or directory
- Org-property-based session scoping for allowed roots, relevant buffers, and
  ignore globs

This is the low-level tool layer used by the Org/gptel workflow.

** [[file:my-gptel-org-workflow.el][my-gptel-org-workflow.el]]

Org-specific structure and conventions for gptel conversations.

It keeps a shallow heading structure so human planning and LLM responses can
coexist cleanly:

- human topic/task headings near the top
- conversation entries below that
- assistant-generated substructure deeper still

It also provides the dynamic system message and reference helpers used during
AI-assisted work.

*** Session scope properties

The gptel Org workflow supports these topic-level properties:

- =GPTEL_ALLOWED_ROOTS=
- =GPTEL_RELEVANT_BUFFERS=
- =GPTEL_IGNORE_GLOBS=
- =GPTEL_INHERIT_DEFAULT_ROOTS=

Property values are whitespace-separated, but entries containing spaces must be
quoted. The parser accepts both single-quoted and double-quoted entries.

Example:

#+begin_src org
:GPTEL_ALLOWED_ROOTS: /Users/me/src/repo-a /Users/me/src/repo-b
:GPTEL_RELEVANT_BUFFERS: pymc_model_refactor.org "*forge: openghg/openghg_inversions #375*" "*magit-diff: ~/repo/*"
:GPTEL_IGNORE_GLOBS: .venv/** data/**
:GPTEL_INHERIT_DEFAULT_ROOTS: nil
#+end_src

If =GPTEL_ALLOWED_ROOTS= is present and =GPTEL_INHERIT_DEFAULT_ROOTS= is absent,
the session uses the explicit roots only. Set =GPTEL_INHERIT_DEFAULT_ROOTS= to a
truthy value to merge the explicit roots with the default/project roots.

Helpful commands:

- =M-x my/gptel-add-session-allowed-roots= appends roots without replacing the property.
- =M-x my/gptel-add-attachment-dir-to-session-allowed-roots= adds the current heading's existing Org attachment directory; with prefix, it creates the directory first.
- =M-x my/gptel-cleanup-response-headings= explicitly repairs and normalizes headings in a selected saved-chat region or the current subtree.

* GitHub Links

** [[file:org-gh/org-gh.el][org-gh/org-gh.el]]

This adds =gh:= Org links for GitHub issues and PRs. It is separate from the
task workflow; the task workflow currently uses the =gh= CLI directly for issue

* Loading

The local modules are made available by adding this directory to Doom's load
path in =config.el= and requiring the relevant files.

At the moment, =my-task-workflow.el= is loaded alongside the existing gptel
modules.
